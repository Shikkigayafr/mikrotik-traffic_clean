## ⚠️ Проблема 1: Неправильные данные в Zabbix (доверие трендам)

### Симптомы:
- В Zabbix графики показывали красивую картинку, но при проверке на MikroTik цифры не совпадали  
- Тренды Zabbix усредняли значения, и пиковые нагрузки терялись  
- Нельзя было увидеть реальный чистый интернет-трафик без L2TP

### Причина:
Zabbix хранит тренды с усреднением. Для точных данных по трафику (особенно если считать дельты и вычитать L2TP) они не подходят.

### Решение:
Перешёл на **прямой сбор через SNMP** — каждые 10 минут забираю сырые значения счётчиков с роутера.

---

## ⚠️ Проблема 2: Счётчики MikroTik хранятся в оперативной памяти

### Симптомы:
- После перезагрузки роутера все значения счётчиков сбрасывались
- Дневные суммы становились меньше реальных
- Данные «пропадали» из статистики

### Причина:
На MikroTik RB760iGS счётчики `ifInOctets` и `ifOutOctets` хранятся в **RAM**. После ребута они обнуляются.

### Решение:
Реализовал **механизм state-файлов** — скрипт сохраняет предыдущие значения, а при новом замере сравнивает их с текущими. Если новое значение меньше старого — считает, что произошла перезагрузка, и корректно рассчитывает дельту (с учётом оборота).

---

## ⚠️ Проблема 3: 32-битные счётчики переполняются

### Симптомы:
- Внезапные «скачки» трафика до нуля на графиках
- Огромные отрицательные значения в логах скрипта
- Дневные отчёты показывали абсурдные цифры

### Причина:
Счётчики на MikroTik — 32-битные, максимум `4 294 967 295` байт (~4 ГБ). Если за 10 минут через интерфейс проходит больше — счётчик обнуляется.

### Решение:
Добавил в скрипт **обработку переполнения**:

```bash
if [ "$WAN_ECM" -ge "$old_wan_ecm" ]; then
    wan_diff=$((WAN_ECM - old_wan_@))
else
    wan_diff=$((4294967295 - old_wan_@ + WAN_@))
fi
```
## ⚠️ Проблема 4: Трафик L2TP идёт через WAN (нужно вычитать)

### Симптомы:
- В Zabbix показывало нагрузку на WAN, но часть трафика уходила в L2TP-туннель
- Невозможно было оценить реальное потребление интернета

### Причина:
L2TP-трафик идёт через WAN-интерфейс. Чтобы увидеть «чистый интернет», нужно вычесть L2TP из общего WAN-трафика.

### Решение:
Научил скрипт **автоматически определять индекс L2TP-интерфейса** по имени и вычитать его из WAN:

```bash
clean_@=$((wan_@ - l2tp_@))
clean_@@=$((wan_@@ - l2tp_@@))
```
## ⚠️ Проблема 5: Потеря данных при недоступности роутера

### Симптомы:
- Если роутер был недоступен в момент замера, скрипт записывал в отчёт нулевые значения
- Дневные суммы занижались
- В логах появлялись ошибки, но скрипт продолжал работать с `0`

### Причина:
Скрипт не проверял, вернул ли `snmpget` значение, и подставлял `0` в расчёты. Это приводило к тому, что при пропущенном замере день становился "короче" по трафику.

### Решение:
Ввёл защиту: если `snmpget` вернул пустую строку (роутер не ответил), данные за этот интервал не учитываются:

```bash
if [ -n "$WAN_@" ] && [ -n "$old_wan_@" ]; then
    # считаем дельту
else
    wan_diff=0
fi
Если хотя бы одно из значений пустое, скрипт пропускает замер, и дневная статистика не искажается.


```

## ✅ Чек-лист: если скрипт не работает

Пройдись по пунктам, если что-то пошло не так:

- [ ] SNMP на роутере включён (проверить через `snmpget -v2c -c public 192.168.1.1 1.3.6.1.2.1.1.1.0`)
- [ ] Community-строка совпадает с той, что указана в скрипте (по умолчанию `public`)
- [ ] Роутер доступен по сети (`ping 192.168.1.1`)
- [ ] Имена L2TP-интерфейсов в скрипте совпадают с реальными (проверить через `snmpwalk`)
- [ ] Есть доступ на запись в папки `snmp_state` и `snmp_data`
- [ ] Утилита `zabbix_sender` установлена на сервере (`which zabbix_sender`)
- [ ] Cron настроен на запуск скрипта каждые 10 минут (`crontab -l`)
- [ ] Логи пишутся в файл и в них нет критических ошибок (`tail -f /home/user/logs/snmp_collector.log`)
- [ ] Скрипты имеют права на выполнение (`chmod +x`)
- [ ] В Zabbix создан хост с именем, указанным в `zabbix_sender` (`-s "Mikrotik-RB760iGS-@"`)
